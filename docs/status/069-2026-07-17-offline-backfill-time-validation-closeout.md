# 069 - 离线全量补抓与 Dashboard 时间边界收口

更新时间：2026-07-17 22:39（Asia/Shanghai）

当前分支：`main`

## 事故与根因

2026-07-16 关机离线后，本地历史库出现 `2026-07-16 07:37:52` 至 `2026-07-17 03:05:02` 的实际缺口。原自动恢复有两个独立截断点：

1. `CATCHUP_MAX_HOURS=24` 会直接丢弃更早的离线窗口。
2. 单次 `catch_up_window` 的全局 `CATCHUP_MAX_STORE=1000` 会从窗口末端向前达到上限，留下窗口前半段缺口，却仍返回 `ok=True`。

金十原始 REST 只读复核确认 `2026-07-16 15:00:00` 至 `17:00:00` 有 193 条消息，因此 Dashboard 空证据不是自然消息空窗，而是本地历史缺失。

## 生产恢复结果

- 使用 30 分钟子窗口补抓完整实际缺口。
- 新增入库：1278 条。
- 缺口窗口现有总数：1279 条，包含原先已存在的结束边界记录。
- 用户指定的 15:00-17:00：193 条，与原始 REST 复核一致。
- `delivery_log` 未因历史补抓写入；补抓发送候选强制为 0。
- `last_ingested_at` 在补抓期间继续由实时消息单调推进，没有被历史记录倒退。
- 手动补抓断点在完整成功后已清空。

## 新的恢复语义

### 自动启动补抓

- 从启动前快照的 `last_ingested_at` 减去保护缓冲开始，一直覆盖到本次启动时间，不再按小时截断。
- 默认使用 `AUTO_CATCHUP_WINDOW_MINUTES=60` 分片，实时 WebSocket 主路仍先启动，补抓在后台执行。
- 自动补抓使用独立 `auto_catch_up_checkpoint*` 状态，不与人工补抓断点相互覆盖。
- REST 失败或子窗口达到 `CATCHUP_MAX_STORE` 时，该窗口不算完成，连续成功游标不会越过缺口；下次启动从旧断点继续。
- 历史记录全部入库；Telegram 不逐条补发。只有完整自动恢复后，才沿用既有 `CATCHUP_TELEGRAM` 单摘要和 gap cooldown 规则。

### 任意时间人工回溯

- `jin10_monitor.py --catch-up --from ... --to ...` 支持任意历史窗口。
- 默认 60 分钟分片、幂等入库、断点续传、不发送 Telegram。
- 只有显式 `--catch-up-telegram` 才启用既有优先级筛选和发送上限。
- `--to` 不允许晚于当前时间。

Dashboard 仍保持业务历史库只读，不提供网页写库端点；这是为了避免浏览器误操作触发大范围金十 REST 或历史 Telegram。

## Dashboard 时间校验

已覆盖仓库中全部四个 `datetime-local` 输入：

- 分析窗口开始 / 结束。
- 详情页行情窗口开始 / 结束。

共同规则：

- 两端必填且格式有效。
- 开始、结束均不能晚于当前时间。
- 开始必须严格早于结束。
- HTML `max` 每 30 秒更新，修改时即时校验，表单提交和行情请求前再次校验。
- 服务端使用同一语义强制校验，绕过浏览器也不能提交非法时间。

性能边界：

- 单次分析窗口最长 7 天，避免把大量历史行全部载入 evidence scoring。
- 行情窗口继续由 interval 对应的 1000 根 K 线上限约束；非法时间在 adapter 查询前返回 400，不发外部请求。
