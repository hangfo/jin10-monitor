## 已采纳

### P0 监控可靠性

采纳并调整实现：

- 健康心跳启动后最多等待 60 秒发送一次 `🟢 [启动]` 首次心跳，消除重启后默认 6 小时静默盲区。
- `last_health_heartbeat_at` 仅在 Telegram 心跳发送成功时更新；失败只记录 `telegram_delivery_status(mode='health_heartbeat')`，避免 Dashboard 把失败心跳误读为成功。
- `health_heartbeat_loop()` 内部隔离普通异常，仅放行 `asyncio.CancelledError`；心跳异常不会向上传播并终止 WebSocket / REST 主路。

未采纳原 review 中的 `asyncio.gather(..., return_exceptions=True)` 写法。原因：当前任务里有多个无限循环，`return_exceptions=True` 会等其他长期任务结束后才返回，无法及时记录某个任务的异常；同时可能掩盖主路失败。当前更优方案是只隔离新增心跳任务自身，主路仍保持异常显式暴露并由 launchd 托底。

### /system 日志诊断增强

采纳：

- `query_recent_monitor_log_events()` 增加 Python 异常类名匹配，覆盖 `RuntimeError:`、`ClientConnectorError(...)`、`asyncio.TimeoutError` 等裸异常名。
- Traceback 块自动向后聚合最多 5 行，直接展示实际异常类型。
- 返回 `ts`、`file_size_kb`、`last_modified` 元信息。
- 增加 30 秒 TTL 缓存，避免每次 `/system` 都读日志尾部。
- 新增只读 JSON 端点 `/api/system/log-events`，支持 `force=true` 清缓存后重新扫描。
- `/system` 日志面板增加时间列、文件元信息、Traceback 高亮和“刷新”按钮。
- REST 403 退避剩余时间增加前端倒计时。

本轮对候选实现做了两点修正：

- 日志时间戳兼容当前真实日志的 `HH:MM:SS [LEVEL]` 格式，不只支持 `YYYY-MM-DD HH:MM:SS`。
- API 测试不引入 `httpx` / `TestClient` 依赖，改为直接调用 FastAPI route endpoint。

### /aggregation 只读统计增强

采纳：

- `query_aggregation_report()` 增加 `skipped_24h` 和 00-23 小时粒度 `hourly_counts`。

该项只读展示本地 SQLite 已有 skipped 诊断，不改变默认聚合开关，不触发 Telegram 或 REST。

理由：

- 它会实际调用 Gemini / OpenAI / compatible / Anthropic Provider API，属于外部请求和可能产生成本的能力。
