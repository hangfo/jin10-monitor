## 2. 本轮新增能力

### 2.1 统一快讯时间显示

Telegram、终端和补拉解析现在统一通过 `item_datetime()` / `item_timestamp()` 处理金十时间字段。

支持两类来源：

- WebSocket：Unix 秒或毫秒时间戳。
- REST/补拉：`YYYY-MM-DD HH:MM:SS` 或 `YYYY-MM-DD HH:MM` 字符串。

统一输出：

```text
YYYY-MM-DD HH:MM:SS
```

意义：

- 避免 WS 显示短时间、REST 显示完整时间造成误判。
- 方便判断消息发生时间、推送时间和补拉窗口。

### 2.2 长停顿自愈补拉

新增配置：

```bash
AUTO_CATCHUP_GAP_SECONDS=300
```

常驻进程运行中，如果 Mac 睡眠、网络长时间断开或进程被系统暂停，REST 轮询恢复后会检测实际停顿时长。

当停顿超过 `AUTO_CATCHUP_GAP_SECONDS`：

1. 读取 SQLite `runtime_state.last_ingested_at`。
2. 补拉 `(last_ingested_at, 恢复时刻]`。
3. 只入库历史消息。
4. 只发送一条 Telegram 摘要。
5. 不逐条补发历史消息。
6. 将补拉窗口内的消息 ID 预热到 `seen_ids`，避免 REST 恢复后重复逐条推送。

设为 `0` 可关闭：

```bash
AUTO_CATCHUP_GAP_SECONDS=0
```

### 2.3 补拉摘要可读性增强

自动补拉摘要现在会列出最多 10 条 T3/T2 重点标题。

示例：

```text
金十自愈补拉完成
窗口：2026-05-09 04:15:00 -> 2026-05-09 07:22:00
入库：42 条
命中推送条件：8 条
分级：⚡ 3 / 🚨 5 / 📰 0

重点消息：
1. ⚡ 2026-05-09 07:20:00 美联储官员表示...
2. 🚨 2026-05-09 06:48:00 伊朗...
```

这样默认不刷屏，但能快速判断是否需要手动补发。

## 3. 当前策略边界

自动补拉仍然坚持：

- 自动入库：开启。
- 自动发摘要：开启。
- 自动逐条补发历史消息：关闭。

原因：

- Telegram 无法按金十事件时间重排消息。
- 大量补发会和实时消息混在一起。
- 逐条补发更适合手动命令触发。

手动补发仍可用：

```bash
python jin10_monitor.py --catch-up \
  --from "2026-05-09 04:15" \
  --to "2026-05-09 07:22" \
  --catch-up-telegram \
  --catch-up-max-send 10
```

## 6. 下一步建议

### P1：延迟提示

只在推送延迟超过阈值时显示，例如：

```text
延迟：123s
```

建议配置：

```bash
SHOW_DELAY_IF_SECONDS=60
```

### P2：AUTO_CATCHUP_SEND 配置化

默认仍不逐条补发，但可以预留手动打开：

```bash
AUTO_CATCHUP_SEND_LEVELS=T3_IMPORTANT,T2_HIGH
AUTO_CATCHUP_MAX_SEND=0
```

设为 `10` 时才补发最多 10 条重点消息。

### P3：事件聚合防刷屏 V2

目标：

- 同主题、同来源或相似标题短时间内合并。
- T3 重要消息保持独立优先。
- 减少 Telegram 重复信息，但完整入库。

### P4：VPS/systemd

本地稳定后再迁移。迁移前需要设计：

- `.env` 安全迁移。
- SQLite 历史库是否同步。
- systemd service。
- 日志轮转。
- 网络连通性与 Telegram 可达性。
